Features

核心功能模块

9 大核心模块覆盖从 Agent 接入到产物交付的完整链路。

🔌

Agent 提供者抽象层

AbstractProvider 接口（6 个核心方法 + capabilities 声明）+ ProviderFactory 注册表。Claude Code SDK、OpenCode CLI、Test 三种 Provider，新增 Provider 只需实现接口并注册，路由/会话/调度层零改动。每个 Provider 有独立的 EventParser，将原生事件格式转换为统一的 UnifiedAgentEvent。

工厂模式接口抽象事件规范化
🐳

Docker 沙箱隔离

每 Session 一个独立容器，三层挂载体系：/sandbox（Agent 运行时）→ /workspace（用户代码）→ /home/agents（持久化身份跨 Session 共享）。Dockerode 管理容器生命周期，execStream 实现 Docker Multiplex 协议解析，毫秒级流式输出推送。Solo 512MB / Group 2GB 内存限制。

容器隔离Multiplex 协议三层挂载
📊

任务编排与 DAG 调度

Planner 拆解需求 → Zod 校验 DAG 计划 → Dispatcher 事件驱动调度（consumeReadyTasks）→ 并行执行无依赖任务 → 依赖满足自动触发下游。三级恢复链：自动重试（指数退避）→ ManagerLoop 审查（LLM 驱动的失败分析）→ Planner 升级（重规划）。全局 + Session 两级并发控制。

DAG 调度事件驱动三级恢复
🧠

ContextBus 共享总线

结构化键值存储（Map + 版本号 + 类型标签），7 天线性衰减 + 引用加权的优先级排序算法。ContextTokenBudget 自动裁剪超出窗口预算的低优先级条目。Pinned 消息机制支持用户手动标记关键信息强制注入。增量查询（getNew）减少重复 Token 消耗。Persisted to PostgreSQL.

优先级衰减增量查询Pinned 注入
🔐

权限代理与安全模型

三种权限模式（ask/smart/trust）适配不同安全场景。PermissionProxy 拦截 Agent 危险操作，基于白名单 + 路径前缀校验判断。ask 模式下通过 WebSocket 推送权限请求给用户实时确认。超时自动拒绝（默认 3 分钟可配）。AES-256-GCM 加密 Provider API Key，解密后注入容器环境变量不经过前端。

纵深防御白名单校验加密存储
🎨

产物预览与引用编辑

HTML 内嵌预览（iframe + 内置静态服务器）、PPTX Canvas 渲染、Monaco DiffEditor 代码对比、结构化 Hunk 接受/拒绝。引用编辑系统：选中 Agent 产物文本 → QuoteToolbar 弹出 → 创建 QuoteReference（含源消息/选择文本/文件路径/行号）→ Agent 精确定位修改。

Deep Dive

核心模块实现细节

点击切换查看各核心模块的详细实现。

1

统一接入接口

定义了 6 个核心行为：启动 Agent、发送提示词、停止执行、注册事件回调、健康检查、切换信任模式。同时声明每个 Provider 的能力矩阵——是否支持持久会话、权限代理、流式输出、独立记忆、独立配置，上层调度模块依据这些能力做出差异化处理。

2

工厂注册机制

使用注册表模式管理所有 Provider——系统启动时注册内置 Provider 的工厂函数，运行时根据名称动态创建实例。上层模块通过统一的创建接口获取 Provider，完全不感知具体实现。新增 Provider 只需实现接口并注册一行代码，其余模块零改动。

3

Claude Code 接入

在 Docker 沙箱容器内以子进程方式启动 Claude 命令行工具，通过容器执行接口的标准输入管道传入提示词。构建安全的运行环境变量集合（包含认证令牌等敏感信息），通过容器环境注入而非文件系统传递。标准输出采用结构化 JSON 流式格式，经事件解析器实时转换为统一事件格式后推送给前端。

4

OpenCode 接入

原生支持会话标识参数，天然具备跨多次调用的上下文保持能力，无需额外维护会话状态。拥有独立的事件解析器处理其特有的输出格式差异。通过跳过权限检查的标志位实现权限代理——危险操作不会阻塞等待，而是将决策权交还给 AgentHub 平台。支持运行时动态切换模型和 API 端点，无需重启。

🐳

容器生命周期

创建阶段：在主机上建立沙箱目录和临时文件目录，通过 Docker 引擎 API 创建容器实例，配置绑定挂载路径和内存上限。启动阶段：容器进入运行状态，等待后续的命令执行。执行阶段：通过容器执行接口在运行中的容器内启动 Agent 进程，实时获取标准输出和标准错误流。销毁阶段：停止并删除容器，可选清理主机上的关联目录。容器命名规则为固定前缀加会话标识，便于运维追踪。

📂

三层挂载体系

第一层——沙箱目录挂载到容器内的沙箱路径，存放 Agent 运行时配置文件、执行计划、临时日志等。第二层——用户工作区目录挂载到容器内的工作区路径，Agent 在此读写项目代码和生成产物。第三层——Agent 持久化身份目录挂载到容器内的共享路径，包含 Agent 的行为配置、长期记忆、自定义技能定义，跨不同会话自动共享，Agent 在一个会话中学到的经验在下一个会话中自动可用。

📡

流式输出机制

Docker 引擎将标准输出和标准错误交织在同一个数据流中传输，采用多路复用帧协议——每帧包含 8 字节头部（标识流类型和数据长度）和数据体。系统内部实现了完整的多路分解逻辑：持续从流中读取帧头、识别流类型（标准输入/标准输出/标准错误）、按声明长度拉取数据体、根据流类型将内容分发到对应的输出回调。这使得 Agent 的实时输出可以在毫秒级延迟内呈现在前端界面上。

🛡️

安全边界

文件系统隔离（容器内无法访问主机 /etc、/home、/root）+ 网络隔离（Docker 桥接 + NAT）+ 资源限制（Solo 512MB / Group 2GB）+ 进程隔离（Linux Namespaces + Capabilities 限制）。

1

计划生成机制

专门的规划 Agent 接收用户自然语言需求、当前会话的可用 Agent 列表（含名称、描述、专长）和项目基础上下文后，将需求拆解为结构化的有向无环图计划。每个任务节点包含唯一标识、标题、描述、执行 Agent 类型、依赖的前置任务列表、预期产物路径。生成的计划先经过严格的 Schema 校验确保结构合法性，随后持久化到数据库的计划执行表中，支持重启恢复。

2

依赖感知调度

调度器持续扫描任务列表，找出所有前置依赖已全部完成的任务，将其标记为就绪状态。然后计算当前可用槽位——取全局并发上限减去当前运行数、以及单会话并发上限减去该会话当前运行数，两者中较小的值作为可启动数量。从就绪任务中按先进先出顺序选取相应数量的任务启动执行。整个调度是事件驱动的：每当一个任务完成或失败，立即触发新一轮扫描以解锁可能的下游任务。

3

三级恢复策略

第一级——自动重试：最多 3 次，采用 2 秒、4 秒、8 秒指数退避间隔，适用于网络抖动等临时性故障。第二级——管理循环审查：独立于执行 Agent 的审查 Agent 分析失败任务的输出、上游结果和当前文件树，判断是表面报错还是实质性失败，做出继续执行、重新规划或终止计划的决策。第三级——规划 Agent 升级：将完整失败上下文提交回规划 Agent，由其重新评估方案可行性，必要时重新设计任务分配或技术路线。

4

两级并发控制

全局级——限制整个系统同时运行的 Agent 总数（默认 5 个，可配置范围 1 到 20），主要目的是控制 AI 服务 API 的并发调用量，避免触发限流。会话级——限制单个会话内同时运行的 Agent 数量（默认 8 个，可配置范围 1 到 50），防止单用户独占所有计算资源。所有并发参数支持运行时通过 API 动态调整并持久化到数据库，运维人员无需重启服务即可根据实际负载调整策略。

1

数据结构设计

内部使用键值对映射存储上下文条目。每个条目包含：唯一键名、任意类型的值、类型标签（区分约定、决策、已知问题、依赖映射、任务交接、项目事实、产物信息共 7 种）、版本号（用于冲突检测和增量查询）、写入者标识、标签数组（用于分类检索）、状态标记（活跃、已废弃、已被替代）、创建和更新时间戳（用于衰减计算）、引用计数（用于优先级加权）。

2

优先级权重与衰减

每条上下文在注入 Agent 前会经过优先级排序，综合三个维度打分。类型基础分反映了不同类别信息的重要程度——项目约定权重最高（Agent 必须遵守的规则），架构决策次之（影响设计方向），已知问题第三（避免踩坑），依次到产物信息最低。时间衰减采用 7 天线性模型，一周前未更新的条目权重降为零。引用加分机制让被多个 Agent 频繁引用的条目自动提升优先级，确保核心知识不会因时间推移而被挤出上下文窗口。

3

上下文预算管理

AI 模型的上下文窗口有限，不能将所有历史条目全部注入。系统在每次 Agent 启动时按优先级降序排列所有条目，逐条注入到上下文提示词中，当累计 Token 数接近预设预算上限（默认可配）时停止。被裁剪的低优先级条目不会丢失——它们仍保留在共享总线中，后续查询时仍然可见。用户手动固定的消息不受预算裁剪影响，始终强制注入到每个 Agent 的上下文中。

4

持久化与恢复

上下文条目通过数据库表持久化。会话标识与键名构成复合唯一约束，值以字符串形式存储（消费方自行反序列化），类型字段提供分类能力，版本号支持乐观锁和增量查询，标签以 JSON 数组存储支持多维检索。同时建立会话标识和计划标识双索引以加速常见查询。会话重启后，所有上下文从数据库完整恢复，Agent 的协作记忆不丢失。

1

连接管理

WebSocket 服务器附加到 HTTP 服务器上，复用同一端口，简化部署和跨域配置。客户端连接时通过 URL 查询参数携带认证令牌和会话标识。连接建立后，在异步的身份校验和数据库查询完成之前，所有到达的消息会被暂存到缓冲区；一旦会话就绪标志位被设置，缓冲区中的消息立即按序重放到消息处理器。这个设计消除了竞态条件——确保在连接完全初始化之前不会丢失任何消息。

2

消息类型体系

系统定义了十余种 WebSocket 消息类型，覆盖完整的协作流程：聊天消息发送、停止 Agent、权限响应、权限模式切换、计划确认与取消、任务重试与修改、强制完成与失败、重规划请求、冲突重试等。每条消息通过类型字段路由到对应的处理函数——聊天消息进入对话处理器，计划操作进入计划处理器，权限操作进入审批处理器——各模块职责单一、边界清晰。

3

Agent 事件推送

Agent 在沙箱内执行期间，标准输出流经事件解析器实时转换为结构化事件，通过 WebSocket 广播推送到前端。事件类型覆盖完整的执行生命周期：推理过程（展示 Agent 的思考链）、工具调用（显示 Agent 调用什么工具及参数）、工具结果（展示返回值）、权限请求（需要用户确认的操作）、Token 用量更新（实时成本追踪）、子 Agent 启停、文件产物通知、阶段完成标记、以及最终的退出事件（携带退出码）。

4

可靠性保障

前端实现了指数退避重连逻辑——断开后按 1 秒、2 秒、4 秒递增等待时间，最大间隔 30 秒。重连成功后从后端 REST 接口拉取最新消息列表和会话状态，与本地缓存合并以消除断线期间的数据差。服务启动时自动扫描并清理所有状态为"流式输出中"的残留消息——将它们标记为已完成，防止因上次异常退出导致前端永久显示"正在输入"。可选的消息序列号机制用于检测丢失事件并请求重发。

Security

五层纵深防御

每一层都假设外层可能被突破，提供独立的防护能力。

🔒

L1 传输层

生产环境 HTTPS + WSS 全流量 TLS 加密。Nginx 反向代理负责证书管理和 TLS 终止。WebSocket 连接升级后仍保持加密通道。

🔑

L2 认证层

JWT Token 无状态认证（7 天有效期）适用于 WebSocket 长连接场景。bcryptjs cost=10 密码哈希。WebSocket Token 通过 URL 查询参数传递（TLS 保护下安全）。

🛡️

L3 授权层

每路由中间件校验 Token → 提取 userId → 数据查询附加 userId 条件。在数据访问层而非路由层实施授权，杜绝 IDOR 漏洞。Session 所有权验证。

🐳

L4 运行时层

Docker 容器文件系统 + 进程隔离。PermissionProxy 三种模式（ask/smart/trust）。路径遍历防护（path.resolve + startsWith 白名单）。超时自动拒绝。

🔐

L5 数据层

AES-256-GCM 加密 Provider API Key。密钥与 JWT Secret 独立。API Key 解密后注入容器 Env，不经过前端和容器文件系统。启动时脏状态自动清理。
