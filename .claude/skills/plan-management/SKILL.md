# AgentHub 计划管理工作流

计划和实现必须保持同步。Plan 是活文档，不是写完就扔的规划稿。

## 核心原则

### 1. 完成即更新

每完成一个功能阶段，**立即**更新对应 plan 文件的勾选状态。不允许累积到多个阶段一起更新。

```bash
# 找到当前 plan
ls docs/superpowers/plans/

# 编辑 plan，将完成的步骤标记为 done
# - [ ] 实现用户登录  →  - [x] 实现用户登录
```

为什么立即更新：累积更新会遗忘细节，而且让其他协作者（包括未来的自己）无法准确判断进度。

### 2. 分歧先讨论

实施过程中发现 plan 的设计需要调整时，**必须先与用户讨论**，而不是静默偏离。

好的做法：
> "Plan 中描述的是 X 方案，但实际实现时发现 Y 问题。建议改为 Z 方案，因为 [原因]。你觉得呢？"

坏的做法：
> （默默改成 Z 方案，plan 和代码各说各话）

### 3. 改 Plan 再改 Code

确认调整方向后，修改顺序是：
1. **先修改 plan 文件**，使其反映新方案
2. **再执行代码修改**

这确保 plan 始终是真相来源（source of truth），而不是过时的文档。

## Plan 文件结构

Plan 文件位于 `docs/superpowers/plans/`，典型结构：

```markdown
# Feature Name

## 目标
简述功能目标

## 阶段 1: [阶段名]
- [ ] 步骤 1
- [ ] 步骤 2
- [x] 已完成的步骤

## 阶段 2: [阶段名]
- [ ] 步骤 1
```

## 操作指南

### 完成一个阶段后

1. 打开对应的 plan 文件
2. 将已完成的步骤从 `- [ ]` 改为 `- [x]`
3. 如果该阶段所有步骤都已完成，在阶段标题旁添加 `✅`
4. 如果有验收标准，确认满足后标记

### 发现需要调整计划时

1. 停下来，不要继续编码
2. 向用户描述发现的问题和建议的调整方案
3. 获得确认后，先更新 plan 文件
4. 然后按新 plan 继续实施

### 开始新功能前

1. 检查是否已有对应的 plan 文件
2. 如果没有，与用户确认是否需要先创建 plan
3. 如果有，阅读 plan 了解全貌后再开始实施

## 与其他 Skill 的关系

- **code-review-workflow**：审查时会对比 plan 和实现的一致性
- **troubleshooting-sop**：排查时会参考 plan 了解预期行为
- **visual-testing**：验证 UI 实现是否符合 plan 中的描述
