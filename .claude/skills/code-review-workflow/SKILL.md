# AgentHub 代码审查工作流

每完成一个功能板块，必须进行代码审查。审查的目标不是找茬，而是确保改动与计划一致、没有引入回归、代码质量达标。

## 触发时机

- 完成一个功能实现后（自然结束点，不是每写一个文件就审查）
- 用户主动要求审查（`/code-review`、"review 一下"等）
- 准备提交代码前

## Step 1: 收集改动

```bash
# 查看未提交的改动
git status
git diff --stat

# 如果改动已提交，对比最近一次 commit
git diff HEAD~1 --stat
```

明确改动涉及哪些文件、多少行变更。如果改动量太大（>500 行），建议拆分审查。

## Step 2: 找到对应的计划文档

```bash
ls docs/superpowers/plans/
```

找到当前功能对应的 plan 文件。审查的核心参照是 plan 中的需求描述和验收标准。

## Step 3: 执行审查

按优先级检查以下维度：

### Critical（必须修复）
- **安全漏洞**：注入、XSS、未授权访问、敏感信息泄露
- **数据丢失风险**：未处理的错误可能导致数据损坏
- **沙箱逃逸**：容器隔离是否被破坏
- **资源泄漏**：Docker 容器、WebSocket 连接、文件句柄未释放

### Important（应该修复）
- **计划偏离**：实现与 plan 描述不一致的地方
- **类型安全**：`any` 类型滥用、类型断言绕过、共享类型不匹配
- **错误处理**：未捕获的异常、错误信息不明确
- **并发问题**：竞态条件、缺少锁机制、WebSocket 消息乱序

### Suggestion（可以改进）
- **代码清晰度**：命名不准确、逻辑过于复杂
- **重复代码**：可提取的公共逻辑
- **测试覆盖**：缺少边界条件测试

## Step 4: 输出审查报告

格式：

```
## 代码审查报告

**改动范围**：X 个文件，+Y/-Z 行
**对应 Plan**：`docs/superpowers/plans/xxx.md`

### Critical
- [文件:行号] 问题描述 → 修复建议

### Important
- [文件:行号] 问题描述 → 修复建议

### Suggestion
- [文件:行号] 问题描述 → 改进建议

### 总结
整体评价（一句话）+ 是否可以提交
```

## Step 5: 修复并验证

1. 修复 Critical 和 Important 问题
2. 修复后运行类型检查确认无新增错误：
   ```bash
   npx tsc --noEmit -p apps/api/tsconfig.json
   npx tsc --noEmit -p apps/web/tsconfig.json
   ```
3. 如果改动涉及 UI，使用可视化测试验证

## 何时可以跳过

**不要跳过。** 即使改动只有一行，也值得花 30 秒确认它与计划一致。审查的代价远低于线上 bug 的代价。
