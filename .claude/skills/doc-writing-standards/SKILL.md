# AgentHub 文档编写规范

基于 Google、Apple、Microsoft、AWS、Stripe、Linear 等大厂文档规范的提炼，适配 AgentHub 项目特点。

## 一、通用铁律（所有文档适用）

### Voice & Tone

| 规则 | 说明 |
|-|-|
| 对话式而非学术腔 | 像懂行的同事在解释，不用"众所周知""显而易见""不难发现" |
| 直接使用"你" | 称呼读者用"你"，不用"用户""开发者""读者" |
| 主动语态 + 现在时 | "AgentRuntime 管理生命周期"而非"生命周期被 AgentRuntime 所管理" |
| 禁止感叹号 | 技术文档不是营销文案 |
| 禁止网络俚语 | tl;dr、ymmv、lol 等一律不用 |
| 不用"请"引导操作 | "点击创建按钮"而非"请点击创建按钮"（Google 明确禁止） |

### 结构与层级

- **标题层级不跳级**：h1→h2→h3，不能 h1→h3
- **每个段落只表达一个概念**：段落首句给出核心信息
- **10 秒可扫读**：读者扫读 10 秒应能理解文档骨架
- **表格优先于段落列表**：对比/配置/参数类信息用表格
- **代码块标注语言**：` ```typescript ` 而非 ` ``` `

### 术语

- **全文术语统一**：同一个概念全文用一个词
- **缩写首次出现定义全称**："ADR（Architecture Decision Record）"
- **禁止方向性语言**："如上所述""见下文""前述"——不利于跳读和翻译
- **文件名/变量名/命令用 code font**：`` `AgentRuntime.ts` `` 而非 "AgentRuntime.ts"

### Callout 使用规范

飞书文档的高亮块（callout）用于标注特殊信息：

| Callout 类型 | emoji | 用途 | 示例 |
|-|-|-|-|
| 核心概念 | 💡 | 定义关键概念 | "AgentHub 是智能中枢（Smart Hub）——主动协调..." |
| 设计决策 | 📌 | 标记关键设计选择 | "采用 KV 黑板模式而非消息队列，因为..." |
| 关键指标 | 🎯 | 标记性能/质量目标 | "从打开页面到首次对话 ≤ 2 分钟" |
| 注意事项 | ⚠️ | 风险、限制、边界 | "当前单机部署，未来需水平扩展时..." |
| 摘要 | 💬 | 章节内容一句话总结 | "涵盖了单聊群聊、任务编排..." |

- 每个 h2 章节最多 3 个 callout，不要过度使用
- callout 内容保持 1-3 句话，不要塞入多段落

## 二、技术架构文档规范

### 章节结构模板

```text
一、项目概述（Overview）
  ├── 1.1 设计哲学（一句话核心定位）
  ├── 1.2 核心概念（理解后续内容必须知道的 3-5 个概念）
  └── 1.3 项目结构（关键目录 + 技术栈一句话）

二、核心架构设计（Architecture）
  ├── 2.X 整体分层架构（图文并茂，图必配文字说明）
  ├── 2.X 请求处理流水线（数据流/控制流）
  ├── 2.X 核心模块详解（按职责拆分，每个模块独立 h3）
  └── 2.X 目录系统设计（文件系统层面的架构）

三、核心功能技术细节（Core Flows）
  按功能拆分 h2，每个功能包含：
  ├── 概念说明（是什么）
  ├── 数据流/流程（怎么工作，优先用 ASCII 图）
  ├── 关键数据结构（TypeScript 接口/类型）
  └── 设计决策（为什么这样做）

四、架构性能优化（Performance & Reliability）
  包含：压缩策略、内存管理、重连机制、容器复用……

五、数据模型设计（Data Model）
  ├── 核心实体表（一个表列出所有模型）
  ├── 实体关系（1:N N:N 关系描述）
  └── API 端点概览（路由 + 方法 + 说明表）

六、会话上下文管理体系（可选，按需独立成章）
```

### 每个设计决策的记录格式（ADR 风格）

```markdown
### 决策：XXX vs YYY

| 维度 | 方案 A（✅ 采用） | 方案 B |
|-|-|-|
| 维度 1 | ... | ... |
| 维度 2 | ... | ... |

**决策**：（一句话说明选择）
**后果**：（选择带来的影响，好处和代价都写）
```

### 技术文档特有规范

- **架构图必须能独立理解**：不看正文也能从图看懂结构
- **分层描述遵循"概念→组件→交互"顺序**：先说什么是什么→再说由什么组成→最后说怎么协作
- **数据流用 ASCII 图而非文字描述**：能让读者一眼看懂流向
- **接口/类型定义直接贴代码**：不要用文字描述接口，直接给 TypeScript
- **每个 h2 章节结尾一句总结**：用 callout 标注，方便扫读时确认是否要深入阅读

## 三、产品设计文档规范

### 章节结构模板

```text
一、产品定位与差异化（Positioning）
  ├── 1.1 定位陈述（一句话）
  ├── 1.2 竞品差异矩阵（表格对比）
  ├── 1.3 关键边界（负责什么 / 不负责什么）
  └── 1.4 核心功能亮点（按需扩展）

二、用户旅程地图（User Journeys）
  每个旅程一张表格：
  ├── 阶段（序号 + 动词短语）
  ├── 用户操作（用户做了什么）
  ├── 系统响应（系统如何反应）
  ├── 触点（涉及的组件/页面）
  └── 关键指标（可选，用 callout 标注）

三、信息架构（Information Architecture）
  ├── 布局说明（三栏/两栏/...）
  ├── 面板系统层级（Tab→子组件→功能→触发条件 表）
  └── 路由结构

四、交互设计规范（Interaction Design）
  按交互模式拆分 h2：
  ├── 指令系统（@ / / 等）
  ├── 富交互（卡片类型、消息气泡）
  ├── 引用机制
  └── 流式渲染

五、关键设计决策记录（ADR）
  格式同技术文档 ADR

六、视觉设计系统（Visual Design）
  ├── 主题系统
  ├── 组件库映射
  ├── 动效规范
  └── 配色语义

七、功能状态矩阵（Feature Matrix）
  按阶段整理的功能实现状态表
```

### 产品文档特有规范

- **旅程描述用表格而非段落**：阶段/操作/响应/触点四列，一目了然
- **设计意图用 callout 标注**："**设计意图**：xxx"，解释为什么这样设计而非仅仅描述是什么
- **竞品矩阵必须真实可验证**：每个 ✅/❌ 要有依据，不能凭感觉写
- **功能状态矩阵诚实标注**：✅ 已实现 / 🔧 部分实现 / 🔜 规划中 / ❌ 不做
- **避免过度营销语言**：产品文档面向内部团队，不是对外销售材料

## 四、AgentHub 项目特有规范

### 代码引用格式

引用代码位置时使用标准格式：
```text
`文件名.ts:行号` — 功能描述
```

示例：`AgentRuntime.ts:315` — Planner 角色识别逻辑

### 章节编号规范

- 技术文档：`一、二、三...` + `1.1、1.2...` + `① ② ③...`
- 产品文档：`一、二、三...` + `1.1、1.2...`
- h2 下至少要有 2 个 h3，否则不拆 h3
- 新增章节插入到逻辑位置，不用 "4.X" 这种临时编号

### AgentHub 术语表

| 术语 | 含义 | 不可替代为 |
|-|-|-|
| Agent | AI 编程代理实体 | AI、机器人、Bot |
| Session / 会话 | 一次对话的上下文容器 | 对话、聊天 |
| Provider | Agent 底层执行引擎抽象 | 平台、后端 |
| Smart Hub / 智能中枢 | AgentHub 的产品定位 | 平台、系统 |
| Sandbox / 沙箱 | Docker 容器隔离环境 | 容器（沙箱更精确） |
| ContextBus | 会话级共享上下文总线 | 上下文管理器 |
| Inbox | Agent 间文件系统消息投递 | 消息队列 |
| Planner | 任务规划 Agent | Main Agent |

### 文档更新流水线（飞书）

```bash
# Step 1: 读文档 outline
lark-cli docs +fetch --api-version v2 --doc "URL" --scope outline --max-depth 3 --format json

# Step 2: 读目标章节
lark-cli docs +fetch --api-version v2 --doc "URL" --scope section --start-block-id "xxx" --detail with-ids --format json

# Step 3: 写入新内容
lark-cli docs +update --api-version v2 --doc "URL" \
  --command block_insert_after --block-id "锚点blockID" \
  --doc-format markdown --content - <<'EOF'
...markdown content...
EOF

# Step 4: 更新文档维护日期（str_replace 页脚日期）
```

## 五、文档质量自检清单

编写完成后逐项检查：

- [ ] 读者扫读 10 秒能否理解文档骨架？
- [ ] 每个 h2 下是否有 ≥2 个 h3（或不需要拆分）？
- [ ] 标题层级是否无跳级（h1→h2→h3）？
- [ ] 表格/代码块的标题行是否用 `background-color` 区分？
- [ ] 代码块是否标注了语言？
- [ ] 全文档术语是否统一（同一个概念同一个词）？
- [ ] 是否有方向性语言（"如上""下文"）需要删除？
- [ ] callout 是否过度使用（每 h2 ≤ 3 个）？
- [ ] 设计决策是否记录了 trade-off（选择了什么、放弃了什么、为什么）？
- [ ] 页脚是否更新了最后修改日期？
